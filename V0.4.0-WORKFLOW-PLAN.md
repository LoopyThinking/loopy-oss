# Loopy OSS — v0.4.0 Workflow Revision Plan

**Status:** planning, not yet executed.
**Branch:** `feat/v0.4.0-workflow-revision` (off `main`).
**Audience:** handoff for an agent (Sonnet) that will implement this end-to-end.
**Author:** Jaime, 2026-04-27.

---

## 0. Context — why this plan exists

The skills installed in Cowork (`loopy-bridge`, `loopy-loop-mapper`, `loopy-collab-bridge`, etc.) were written for the SaaS product at `loopythinking.ai`. When used against the OSS instance running on `localhost:3000` they ask for things that don't exist here: a corporate user profile, the Orion Registry, and unified team management.

OSS already has the building blocks (database tables, REST endpoints), but the UI and the skills are not aligned. This plan closes the four gaps below.

| # | Gap | Backend status | UI status |
|---|-----|----------------|-----------|
| 1 | Skills point to Orion / Cloud profile | n/a (out of OSS) | n/a |
| 2 | Team / member management | done (`/orgs/:id/members`, `/orgs/:id/invites`) | **missing** |
| 3 | Skills/tools per agent visibility | done (`/agents/:id/skills`, `/tools`) | **missing** |
| 4 | Loops del equipo (admin view of others' loops) | partial — `/admin/loops` aggregated only | **missing** detail view + dashboard toggle |

There is also a small **bug**: the sidebar in `Layout.tsx` links to `/loops`, but `App.tsx` has no route for `/loops` — the catch-all sends users back to `/dashboard`, which is confusing.

---

## 1. Architectural decisions (lock these in before coding)

1. **Naming.** No "Orion" anywhere in OSS. The agent registry table is `agent_registry`; surfaces to the user as "Agentes". Keep this consistent.
2. **Versioning.** This is a 0.4.0 release. Bump:
   - `apps/web/package.json` and `apps/api/package.json` to `0.4.0`.
   - The footer in `Layout.tsx` (`v0.3.0` → `v0.4.0`).
3. **Schema additions.** One new migration: `009_v040_workflow.sql` (see §6.1). No breaking changes to existing tables — only `ALTER TABLE` adds.
4. **Authorization model.** A loop is visible to:
   - its `user_id` (owner), always; or
   - any `org_member` with role `admin` or `owner` of the loop's `org_id`.
   This is enforced server-side in `loops.ts` — never trust the client.
5. **Skills strategy.** Do **not** modify the existing cloud-targeted skills under `~/.claude/skills/`. Ship a new set of OSS-specific skills inside the OSS repo at `packages/cowork-plugin/`, packaged as a Cowork `.plugin` bundle. The user installs the OSS plugin alongside (or instead of) the cloud one. This keeps the cloud skills untouched and gives OSS users a clean, working surface.
6. **No email transport.** Invites remain link-based (operator copies and delivers). This is an explicit OSS principle — keep it.
7. **Tests.** Every API route added/modified gets at least one happy-path + one auth-failure test. UI gets smoke render tests.

## 2. Block A — Adapt skills for OSS (`packages/cowork-plugin/`)

### 2.1 Goal

A user installing the OSS Cowork plugin should be able, from a chat, to:
- Register an agent and persist its token (`POST /agents`).
- Create a loop and emit signals (`POST /loops`, `POST /signals`).
- Register declared skills and tools for that agent (`POST /agents/:id/skills`, `/tools`).
- Generate an org invite link (`POST /orgs/:id/invites`).

…all using `LOOPY_BASE_URL` and `LOOPY_AGENT_TOKEN` from env. No mention of Orion or Cloud-only constructs.

### 2.2 Plugin structure to create

```
packages/cowork-plugin/
├── plugin.json                       # Cowork plugin manifest
├── README.md                         # how to install + env vars
├── skills/
│   ├── loopy-oss-bridge/SKILL.md     # setup, agent registration, profile fill
│   ├── loopy-oss-loop-mapper/SKILL.md # work signal → loop pipeline
│   ├── loopy-oss-signal-emit/SKILL.md # emit a signal to an existing loop
│   ├── loopy-oss-collab-bridge/SKILL.md # invite teammate, share loop
│   └── loopy-oss-ipl-tracker/SKILL.md # IPL minutes / hours summary
└── package.json                      # name: "@loopythinking/cowork-plugin", version 0.4.0
```

Each `SKILL.md` must:
- Open with a `## Activation` block listing trigger phrases (Spanish + English).
- Include a `## Environment` block reading `LOOPY_BASE_URL` (default `http://localhost:3001`) and `LOOPY_AGENT_TOKEN`.
- Document the exact REST calls the agent should make. Provide curl examples.
- End with a `## Failure modes` block: missing token, 401, 403 (org context), 404 (loop not found).

### 2.3 Specific replacements (cloud → OSS)

| Cloud action (existing skill) | OSS replacement |
|-------------------------------|-----------------|
| Register in Orion Registry | `POST {LOOPY_BASE_URL}/agents` body `{ agent_name, description }` → returns `{ id, token }`. Persist token in env. |
| "Complete user profile" (corporate fields) | `GET /me` then `PATCH /me { display_name }`. Nothing more. Skill must NOT ask for role / area / team. |
| Fetch organization context | `GET /orgs` → choose first → set `X-Org-Id` header on subsequent requests. |
| Invite teammate | `POST /orgs/:orgId/invites { role, expires_in_days }` → return `accept_url`. |
| Create loop | `POST /loops { title, hypothesis?, scope }` (`scope ∈ personal|team|organizational`). |
| Emit signal | `POST /signals { loop_id, type, payload? }` (`type ∈ perception|interpretation|decision|action|reflection`). |
| Register skill | `POST /agents/:agentId/skills { skillName, version?, description?, source?, metadata? }`. |
| Register tool | `POST /agents/:agentId/tools { toolName, toolType, description?, metadata? }`. |
| IPL summary | `GET /admin/overview` (admin+) or compute from `GET /loops` + `ipl_minutes` field. |

### 2.4 Acceptance criteria

- `npm run dev` in the OSS repo → install the plugin in Cowork → open chat → say *"crea un loop de prueba"* → loop appears in `/dashboard`.
- Saying *"registra mi agente"* without a token in env triggers the registration flow and stores the token.
- No skill prompts for "rol corporativo", "área", "Orion", or any Cloud-only field.

### 3.3 New API endpoints (small additions to `orgs.ts` / `invites.ts`)

- `GET /orgs/:id/invites` (admin+): list pending invites for the org.
  ```ts
  // Returns: Array<{ id, role, expires_at, created_at, invited_by_email }>
  ```
- `DELETE /orgs/:id/invites/:inviteId` (admin+): revoke an unaccepted invite (set `expires_at = now()` or hard-delete).

- A `member` (not admin) sees a 403 page or is redirected from `/admin/team`.

## 5. Block D — Team loops view

### 5.1 Server-side authorization upgrade

Edit `apps/api/src/routes/loops.ts`. The current implementations are owner-only:

```sql
WHERE id = ${id} AND user_id = ${userId}
```

Replace with an OR clause that also allows admins of the loop's org. Pseudo-implementation pattern:

```ts
// helper
async function canSeeLoop(userId: string, loopId: string): Promise<boolean> {
  const [row] = await sql<Array<{ allowed: boolean }>>`
    SELECT
      (l.user_id = ${userId})
      OR EXISTS (
        SELECT 1 FROM org_members m
        WHERE m.user_id = ${userId}
          AND m.org_id = l.org_id
          AND m.role IN ('admin', 'owner')
      ) AS allowed
    FROM loops l
    WHERE l.id = ${loopId}
  `
  return !!row?.allowed
}
```

Apply this in `GET /loops/:id`, `POST /signals` (signal write should respect ownership rules — admins can read, but only the owner can mutate), and `POST /loops/:id/close` (owner-only — admins should not close someone else's loop without the owner's act).

### 5.2 New query param on `GET /loops`

Today `GET /loops` filters by `user_id = ${userId}`. Add `scope` param:

- `GET /loops` → unchanged behavior (mis loops).
- `GET /loops?scope=team` → admin+ only. Returns all loops where `org_id = currentOrgId` (use `X-Org-Id` header). Non-admins requesting `scope=team` get 403.
- `GET /loops?scope=mine` → explicit alias of default.

Reject unknown `scope` values with 400.

### 5.3 Dashboard UI changes

In `apps/web/src/pages/Dashboard.tsx`:
- Read current org role from `useMe()` or `Layout` context.
- If admin+, render a segmented toggle at the top: `[ Mis loops ] [ Loops del equipo ]`.
- "Loops del equipo" calls `useLoops({ scope: 'team' })`. Each `LoopCard` should additionally show the owner's `display_name` (small caption under the title) so admins can tell whose loop is whose.
- When viewing a team loop, hide the "Cerrar loop" button on `LoopDetail.tsx` if `loop.user_id !== currentUserId`.

### 5.4 Add `/loops` route (fix the dead nav link)

The sidebar in `Layout.tsx` links to `/loops` but `App.tsx` has no such route. Add:

```tsx
<Route path="/loops" element={<RequireAuth><Loops /></RequireAuth>} />
```

`apps/web/src/pages/Loops.tsx` is essentially the Dashboard's "loops table" view but without KPIs — full-width table sortable by status / scope / confidence / IPL. It can share `useLoops`. (Alternatively: redirect `/loops` → `/dashboard`. Pick one — recommended: full table page, since the sidebar already differentiates "Dashboard" from "Loops".)

### 5.5 Differentiate `/admin` so it stops feeling like a duplicate of `/dashboard`

In `Admin.tsx`:
- Add a 5th KPI: **Top owner** — user with the most active loops.
- Add a small breakdown card: **IPL por usuario** — bar list (top 5).
- Group the loops table by owner by default (collapse rows under owner headers). Keep current sort options.
- Add an **Agentes más activos** card: top 3 agents by `total_signals`.

These all use data the API already returns from `/admin/loops` and `/admin/agents`; no new endpoint needed.

### 5.6 Acceptance criteria

- Owner role: `/dashboard` shows toggle "Mis loops / Loops del equipo". Switching to "Loops del equipo" shows loops belonging to other org members. Clicking one opens its detail (read-only — no "Cerrar" button).
- Member role: no toggle visible; `/dashboard` works as today.
- The `/loops` sidebar link no longer redirects to `/dashboard` — it opens the full loops table.
- `/admin` and `/dashboard` look visibly different at a glance: KPIs + per-owner grouping vs. personal card list.

## 6. Database & supporting work

### 6.1 Migration `009_v040_workflow.sql`

Only one schema change is required, but place it in a single migration for traceability:

```sql
-- Migration 009: v0.4.0 workflow

-- (a) Optional: a column to flag whether an invite was revoked vs. just expired.
ALTER TABLE org_invites
  ADD COLUMN IF NOT EXISTS revoked_at TIMESTAMPTZ;

CREATE INDEX IF NOT EXISTS idx_org_invites_pending
  ON org_invites (org_id)
  WHERE accepted_at IS NULL AND revoked_at IS NULL;

-- (b) Convenience view for admin loop browsing — optional, but speeds up the
-- "loops del equipo" query and centralizes the auth predicate.
CREATE OR REPLACE VIEW v_team_loops AS
SELECT
  l.*,
  u.display_name AS owner_name,
  u.email        AS owner_email
FROM loops l
LEFT JOIN users u ON u.id = l.user_id;
```

The DELETE invite endpoint should set `revoked_at = now()` rather than hard-deleting, so invitations remain auditable.

### 6.2 Type changes

In `apps/api/src/types.ts`:
- Add `OrgInvite` interface.
- Extend `Loop` with optional `owner_name`, `owner_email` for the team view.

In `apps/web/src/lib/api.ts`:
- Add `api.orgs.listInvites(orgId)`, `api.orgs.revokeInvite(orgId, inviteId)`.
- Add `api.agents.skills.list(agentId)`, `.deactivate(agentId, skillId)` and equivalents for tools.
- Add scope param to `api.loops.list({ scope?: 'mine' | 'team' })`.

### 6.3 SDK updates (`packages/sdk`)

If we expose new endpoints, mirror them in the SDK (`@loopythinking/sdk` v0.4.0):
- `client.skills.list(agentId)` / `.deactivate()`
- `client.tools.list(agentId)` / `.deactivate()`
- `client.orgs.listInvites(orgId)` / `.revokeInvite()`
- `client.loops.list({ scope })`

Bump `packages/sdk/package.json` to `0.4.0` and add a `CHANGELOG` entry.

## 8. Manual QA checklist (before merging)

Run with two browsers / two users in the same org:

- [ ] User A registers, creates an org, generates an invite link.
- [ ] User B opens the invite link, registers, accepts → appears as `member` in `/admin/team`.
- [ ] User A promotes B to `admin` → B sees the "Loops del equipo" toggle in `/dashboard`.
- [ ] User A creates a personal loop → User B (admin) sees it under "Loops del equipo" with A's name as owner.
- [ ] B opens A's loop → can read it, cannot close it (button hidden / 403 on attempt).
- [ ] User A registers an agent via the OSS plugin in Cowork → appears in `/agents`.
- [ ] User A registers a skill `docx` for that agent via SDK → appears in `/agents/:id` Skills tab.
- [ ] User A deactivates the skill from the UI → row turns gray, `GET /agents/:id/skills` no longer returns it.
- [ ] User A revokes B's invite link before B uses it → public `GET /invites/:token` returns 410.
- [ ] `/loops` sidebar link opens the loops table (not a redirect).
- [ ] `/admin` shows per-owner grouping, top owner KPI, top agents card.

## 9. Out of scope (explicit non-goals for v0.4.0)

- Email transport for invites. Stays link-based.
- Real-time updates (WebSockets / SSE) on dashboards.
- Per-loop ACLs beyond owner / org-admin.
- Skill marketplace / discovery.
- Billing or quotas.

These belong to v0.5.0+ and should not creep into this branch.
